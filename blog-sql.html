SQL Essentials: From Basics to Advanced

Published on Jan 10, 2025  |  Category: SQL

Overview

SQL is the language of relational databases — selecting, filtering, joining, grouping and transforming data. In this post I cover core query constructs, schema design, indexing basics, transactions, and optimization heuristics that I found essential while building data-driven applications.

Core Concepts

Tables & Schemas
Rows and columns model entities; normalization reduces redundancy while denormalization can improve read performance.
Primary & Foreign Keys
Primary keys uniquely identify rows. Foreign keys represent relationships — essential for joins and referential integrity.
Indexes
Indexing speeds lookups but adds write cost and storage overhead. Choose indexes on frequently filtered/sorted columns.
Transactions & ACID
Transactions ensure atomicity and consistency. Understanding isolation levels helps prevent race conditions and read anomalies.

Common Queries & Patterns

Below are practical query patterns I used frequently.

1) Basic SELECT with JOIN 

SELECT u.id, u.name, o.order_count
FROM users u
LEFT JOIN (
  SELECT user_id, COUNT(*) AS order_count
  FROM orders
  GROUP BY user_id
) o ON u.id = o.user_id
WHERE u.is_active = TRUE;
2) Window Functions (ranking) 

SELECT user_id, purchase_amount,
       RANK() OVER (PARTITION BY user_id ORDER BY purchase_amount DESC) AS rnk
FROM purchases;

Schema Design: Lessons

I learned schema design by building sample apps: start normalized, then denormalize only when performance dictates. Maintain good naming conventions, use surrogate keys where appropriate, and document relationships clearly.

Indexing & Performance

Index selectively — high-cardinality columns used for filtering or ordering are good candidates. Use EXPLAIN/EXPLAIN ANALYZE to inspect query plans. Beware of full-table scans caused by functions on indexed columns or poor join orders.

Transactions & Concurrency

Doing concurrency correctly saved me from subtle bugs. Use short transactions, avoid user prompts inside transactions, and choose suitable isolation levels for your use case.

Real Project Example

In a project that tracked user events, I initially stored every event row in a single table — reads became slow. Solution: moved to partitioned tables by month and added composite indexes on (user_id, event_time) — read latency improved drastically while write performance remained acceptable.

Practical Tips

  • Prefer explicit column lists in SELECT rather than `*` for stable queries.
  • Use LIMIT for pagination and OFFSET carefully — keyset pagination is often superior for large datasets.
  • Back up and test schema migrations locally before running in production.

My Learning Experience

Learning SQL enhanced how I design systems. Debugging slow queries taught me to think like the database engine — reducing table scans, streamlining joins, and thinking about index selectivity became routine in my development flow.
